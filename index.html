HTML入门

Veröffentlicht am 2017-03-24

HTML是一种标记语言，（markdown也应该是一种标记语言），标记以供计算机去识别。

开头首先要声明(因为markdown貌似兼容html格式，所以要稍加编辑才显示），声明我写的这篇是html

< !DOCTYPE html>

然后加一个< html>、< /html>, 标示整体区间。

然后要有个头，< head>, < /head>，在其间，加上< title>, < /title>, 就可以显示网页标题了，一般是显示在标签上或者浏览器页面最上方（不在网页正文显示）。

然后自然要有个身体，< body>, < /body>, 中间的内容就会显示在网页内：

  • 要网页显示文章标题，或者是加大号字体，可以用< h1>…< /h1>标示，这里数字1可以一直替换到6，相应的字体是依次变小的。
  • 要添加一个段落，就在段落首尾加上< p>和< /p>就可以。
  • 要加入无序列表，在无序列表首尾加< ul>和< /ul>，然后在每行的无序列表元素的首尾加上< li>, < /li>。就可以显示无序列表的每一项了。
  • 而要加入有序列表，则方法如无序列表，只是把整个列表首尾改成< ol>和< /ol>就可以。

——3-27添加——
加入超链接文字的话，anchor元素：< a href=”http://xxx.xxx.com“ target=”_blank”> Read More < /a> 这样就把Read More加上了超链接的属性了。

加入图片的话，直接使用< img src=”http://xxx.xxx/xxx.jpg“ />
把图片加上超链接属性的话，直接将图片的代码覆盖上面的“Read More”就好了

要在网页上体现空白行，在代码里直接回车是不行的，需要代码< br/>来实现。

要添加注释，或者将已有代码临时转换成注释，而不生效，则可用< !— ABCD —> 这样中间所有的内容（ABCD）全都不会被体现。

CSS : 层叠样式表 Cascading Style Sheet

一般不会将CSS代码与html代码混合写，而是将css代码另起一页，使用链接关联起来。具体办法是，在html的< head>区块内，加入< link>，如果css文件放在了一个网络地址，则加入< link href=”xxx.com/xxx.css” style=”text/css” rel=”stylesheet”>，当然如果css文件与html文件放在了同一个地址，则将上面的网络地址换成相对地址就可以了/xxx.css

17年3月21日杂谈

Veröffentlicht am 2017-03-21

现在公司内如我们一样的底层工程师们，都有一种风气，正如：

个人自扫门前雪，休管他人瓦上霜。

就是当出现问题的时候，大家一起开会讨论，临到定义任务的时候，有人站出来说是谁谁谁的任务，那肯定会有人不高兴，多半会出言争辩，试图推开责任。而且，在出来问题无人可以定义是谁应该牵头时，大家也很少会第一个起来牵头干。

造成这种现象的原因到底是什么呢？先搞清楚大家为啥会推诿。

  • 不愿出头。就是懒吧，不愿干事情，不愿承担更大的责任，甚至是应有的责任。对于这样的心态，于他本人、于公司就都不太好了。于本人是其白白丧失了机会成本，本来可能更有机会让自己更有成就感的，或者说能让自己收获更大、更充实的；而于公司，则显然是没能达到高效运转的状态，最终多少都会反映在公司整体的效益上。
  • 不敢出头。不敢出头，原因可能就比较多了。有的人就是恰好胆小，不敢主动带头推动，而恰巧这个问题就发生在了这种类型的人身边了，也就无人认领；另外就是员工自己没有问题，心里都想去牵头做，但是想想万一做砸了的后果，就退缩了。这种情况其实员工、企业是有点冤枉的，而关键应该在于领导。领导对员工工作的反馈、评价、奖惩的方法和方式，都会对员工的办事方式造成影响的。
  • 另外就是觉得自己手里是没有足够的资源，想带头做也难。这种情况就员工本人来说也是没办法的，唯一能做的就是升级困难，让领导给铺路。但是要是在规章、流程层面就有这种便利的话，那员工自然就没有了这方面的顾虑。

像是今天，楼上一老外就直接给给我派了一个活，一开始没听清说让我分析差异，我心里当时就不愿意了，因为我觉得事情不是这么干的，本来就有专门分析的人，为啥要我去分析，况且我手里根本也没啥资源啊，让我咋干？后来问清楚了，其实要我干的不是我想的那样，也是我可以干的活，那我就接受了，其实也是整个team一起做的事情。

另外对于编程的路线，一直没有想清楚。前面断断续续学的Python语法，其实也没有学完，现在又想建个网站，我有点迷糊了，不知怎么行进。查了查，发现网上各种建议都有。现在初步计划是跟着Codecademy的步骤来试试吧，不知道教程内容怎样。

而对于学习方法，了解到的有两种：

  • 以目标或者效果为导向的学习，这样的学习方法成果会比较明显，最后达成了后自己的印象也会比较深刻。通俗的讲，就是先想清楚自己想干啥，然后直接搜索有哪些到达方法，照着方法做，中途遇到不明白或者缺少的内容，再接着去搜索，直到最终达到自己的目的。当然最后需要做个总结，加入笔记本，效果会更好，也方便自己将来复习。
  • 参照“费曼学习法”。诺奖大师的方法，自然不会差。网上说到这个名词可能也不是他创建的，但是从他的相关资料中也是有迹可循的。主要方法，我目前的理解，就是使用最通俗式的语言，写下来，或者说出来，去教别人怎么做。其实原理应该是很简单的，就是“教就是最好的学习方法”。以大妈式的语言，来讲解给一个小白听，需要思考，需要经常反问自己，小白对我说的能听懂吗？没有有他可能不懂得地方？这个地方能拆解到哪一层，小白能大概率的听懂呢？我这样组织的顺序，小白理解起来OK吗？全部下来，语句组织方面有无继续提升的地方呢？

零零碎碎的文字，也是要践行多多写作这件事。直接就感受到了，写作是整理思维的一种方式啊。

有好几次了，总是说要跟着网上的教程，搭一个自己的博客，这回姑且可以说是达成了吧。

主要元素还是Github Page + Hexo，对于我这样的小白而言，也并不容易。总体感觉如下：

  • 免费的空间、域名。对于我这很种轻量式写者足够用了。
  • 操作还算简单，但是每次都要输入几行代码才能发布上去。而且不是直接发布文字，而是把内容写好后再编辑到post里的md文件上去，略有麻烦。
  • 暂时还没找到用手机直接发文章的办法。不过对于我来说，暂时用不上，能写出文章来就不错了，持续就好，不求多。
  • 感觉只能是在同一台电脑上做，如果换了一台电脑，那就麻烦了。不仅都要设置好环境，可能还需要去研究文件夹同步的问题。

不过这些问题不是重点，持续，才是重点吧。
